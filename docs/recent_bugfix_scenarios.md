# 最近两个 Commit 解决的场景

## `acee051` `fix(plugin): preserve unpacked array suffixes`

这个提交解决的是代码生成阶段会把数组维度或切片信息弄丢的几类问题。

### 场景 1：已经手工改过的实例连接，再次自动例化后被改坏

典型现象：

- 原来手工写成 `.a(a[1])`，再次执行自动例化后可能被回退成 `.a(a)`
- 原来手工写成 `.b(b[7:4])`，再次执行自动例化后可能被错误改成别的默认连接形式

修正后：

- 如果实例连接已经被用户改成带索引、切片或更复杂的表达式，再次执行自动例化时会继续保留这些写法，而不是按默认连接重新覆盖。

### 场景 2：子模块端口带数组后缀时，自动生成的连线声明会丢维度

典型现象：

- 子模块端口本来是带数组后缀的，例如 `foo [1:0]`
- 自动生成连线声明时，只保留了名字，后面的数组后缀没有带出来
- 最终生成的声明比真实端口少一维，甚至被当成普通单信号处理

修正后：

- 自动生成的 wire/logic 声明会把名字后面的数组后缀一起保留下来，声明结果和端口维度保持一致。

### 场景 3：同名自动连线来自多个地方时，位宽可能不稳定

典型现象：

- 同一个自动生成信号在不同连接里出现过不同但兼容的位宽写法
- 生成结果可能只取最后一次看到的宽度，导致声明依赖扫描顺序

修正后：

- 对纯数字位宽会先做归并，再输出更稳定的最终声明，减少“同一份代码偶尔生成不同位宽”的情况。

## `9700142` `udpate: add gitignore and AGENT.md`

这个提交不是插件运行时 bug 修复，主要解决的是仓库维护和协作中的两个高频问题。

### 场景 1：构建产物或临时输出容易误提交

典型现象：

- 本地产生了 `build/` 目录
- 提交时这些产物可能混进版本库，形成无关噪音

修正后：

- `build/` 被加入忽略规则，默认不会再作为正常改动进入提交范围。

### 场景 2：协作时缺少统一约定，容易改错位置或漏做基本验证

典型现象：

- 新参与者或自动化工具不知道入口文件、模块划分和推荐验证方式
- 改动后没有用一致的最小 smoke test 去确认插件能加载
- 提交信息和 PR 描述风格不统一

修正后：

- 仓库内新增了协作文档，明确了目录结构、常用验证命令、Vimscript 风格、手工测试重点以及提交规范，降低了协作时的歧义。
